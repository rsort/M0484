§ 1

Què és Redis?

Redis (Remote Dictionary Server) és una base de dades NoSQL de tipus clau-valor que emmagatzema les dades completament en memòria RAM. Va ser creada per Salvatore Sanfilippo el 2009 i és avui una de les bases de dades més ràpides del món, emprada per empreses com Twitter, GitHub, Stack Overflow i Airbnb.

A diferència de MongoDB (que emmagatzema documents JSON al disc), Redis prioritza la velocitat màxima d'accés emmagatzemant totes les dades directament a la memòria principal.

Característiques principals

  • Emmagatzematge 100% en memòria RAM: accés en microsegons.
  • Suporta múltiples estructures de dades especialitzades (strings, llistes, sets, hashes, etc.).
  • Persistència opcional: pot guardar una còpia de les dades al disc periòdicament.
  • Ideal per a caché, cues de missatgeria, comptadors en temps real i sessions.
  • Open-source i multi-plataforma.
  • 8a posició al rànquing global DB-Engines (maig 2026) amb puntuació de 147,63.

§ 2

Casos d'ús principals

⏱Real-time Data Store

Emmagatzematge de dades en temps real: marcadors en viu, sistemes de votació, anàlisi de fluxos de dades amb latència mínima.

📨Streaming & Messaging

Cues de missatgeria i comunicació entre microserveis. Permet implementar patrons Pub/Sub i processing pipelines.

💾Caching & Session Storage

El cas d'ús més habitual: emmagatzemar resultats de consultes costoses i gestionar sessions d'usuari per servir-les ràpidament.

Exemple típic de caché

EscenariFluxTemps
Sense RedisClient → API → Base de Dades → API → Client~15 s
1a petició amb RedisClient → API → BD → Redis (guarda) → API → Client~15 s
2a+ peticionsClient → API → Redis (serveix) → API → Client~0,5 s

A partir de la 2a petició, Redis serveix les dades directament des de memòria, sense accés HTTP a servidors externs ni consultes SQL. El procés pot ser fins a 30 vegades més ràpid.

§ 3

Estructures de dades

Redis no només guarda dades simples: permet treballar amb 8 estructures de dades especialitzades segons la necessitat del problema.

1. Strings

El tipus més bàsic. Pot emmagatzemar text, números o dades binàries. Màxim 512 MB per valor.

SET nom "Laura"
SET edat 23
GET nom                    // ➜ "Laura"
EXISTS edat                // ➜ 1 (existeix)
STRLEN nom                 // ➜ 5 (caràcters)
DEL nom                    // Elimina la clau

// Exemple JoinApp
SET joinapp:titol "JoinApp - Connecta amb la teva comunitat"

2. Lists (Llistes ordenades)

Llistes de valors ordenats (similar als arrays). Permet afegir elements al principi (LPUSH) o al final (RPUSH). Admeten duplicats.

LPUSH tasques "Estudiar"
RPUSH tasques "Fer exercici Redis"
LRANGE tasques 0 -1        // ➜ ["Estudiar", "Fer exercici Redis"]
LLEN tasques               // ➜ 2
LPOP tasques               // Treu i retorna el primer element
RPOP tasques               // Treu i retorna l'últim element

// Exemple JoinApp: historial d'activitats d'un usuari
RPUSH usuari:42:historial "Ruta pel Montseny"
RPUSH usuari:42:historial "Ciclisme Garraf"

3. Sets (Conjunts)

Col·lecció no ordenada d'elements únics. No permet duplicats. Perfecte per emmagatzemar etiquetes o usuaris únics.

SADD alumnes "Laura"
SADD alumnes "Joan"
SADD alumnes "Laura"       // Ignorat: ja existeix
SMEMBERS alumnes           // ➜ {Laura, Joan}
SCARD alumnes              // ➜ 2
SISMEMBER alumnes "Joan"   // ➜ 1 (existeix)
SREM alumnes "Joan"        // Elimina Joan del set

// Exemple JoinApp: assignatures del cicle (sense duplicats)
SADD cicle:daw:assignatures "Bases de Dades" "Programació" "Sistemes"

4. Sorted Sets (Conjunts ordenats)

Com els sets, però cada element té una puntuació numèrica (score) que defineix el seu ordre. Ideal per a rankings i sistemes de puntuació.

ZADD ranking 100 "Laura"
ZADD ranking 150 "Joan"
ZADD ranking 75  "Maria"
ZRANGE ranking 0 -1 WITHSCORES   // ➜ Maria(75) Laura(100) Joan(150)
ZREVRANGE ranking 0 -1 WITHSCORES // Ordre descendent
ZSCORE ranking "Joan"             // ➜ 150
ZRANK ranking "Laura"             // ➜ 1 (posició, comença per 0)
ZCARD ranking                     // ➜ 3

// Exemple JoinApp: ranking d'activitats per participants
ZADD activitats:ranking 342 "Ruta Montserrat"
ZADD activitats:ranking 127 "Kayak Port Olímpic"

5. Hashes (Diccionaris)

Estructura de clau-valor dins d'una clau. Permet representar objectes (usuaris, productes) de manera similar a un JSON pla. Molt eficient per a entitats amb múltiples camps.

HSET usuari:1 nom "Laura" edat "23" hobby "senderisme"
HGET usuari:1 nom           // ➜ "Laura"
HGETALL usuari:1            // ➜ nom:Laura, edat:23, hobby:senderisme
HKEYS usuari:1              // ➜ nom, edat, hobby
HVALS usuari:1              // ➜ Laura, 23, senderisme
HEXISTS usuari:1 edat       // ➜ 1 (existeix)
HLEN usuari:1               // ➜ 3 (nombre de camps)

// Exemple JoinApp: dades d'un esdeveniment
HSET event:101 nom "Trail Tibidabo" ubicacio "Barcelona" participants "48"

6. Bitmaps

Emmagatzema bits (0 o 1) en posicions específiques. Extremadament eficient en memòria per fer seguiment binari (sí/no) de grans quantitats d'elements.

// Posició = ID alumne, 1=present, 0=absent
SETBIT assistencia:avui 4 1  // Alumne 4 present
SETBIT assistencia:avui 7 1  // Alumne 7 present
GETBIT assistencia:avui 4    // ➜ 1 (present)
GETBIT assistencia:avui 2    // ➜ 0 (absent)
BITCOUNT assistencia:avui    // ➜ 2 (alumnes presents)

7. HyperLogLogs

Estructura probabilística per comptar elements únics usant molt poca memòria. No emmagatzema els elements reals, sinó una estimació del nombre d'elements únics (precisió ~0,81%).

PFADD visites "laura" "joan" "maria"
PFADD visites "laura"          // Duplicat: ignorat
PFADD visites "pau"
PFCOUNT visites                // ➜ 4 (estimació d'únics)

// Exemple JoinApp: usuaris únics connectats
PFADD assignatura:daw:usuaris "anna" "marc" "laura"
PFCOUNT assignatura:daw:usuaris

8. Streams

Estructura de dades per a esdeveniments o missatges ordenats en el temps. Similar a un log append-only: permet que múltiples consumidors llegeixin les dades a ritmes diferents.

XADD logs * event "login" user "anna"
// * = Redis genera un ID automàtic basat en el temps

XRANGE logs - +              // Totes les entrades del stream
XREVRANGE logs + - COUNT 5   // Les últimes 5 entrades
XREAD COUNT 2 STREAMS logs 0 // Llegeix els primers 2
XLEN logs                    // Nombre total d'entrades

// Exemple JoinApp: avisos de l'assignatura
XADD avis:daw * tipus "examen" contingut "MongoDB - dimecres 10:00"
XRANGE avis:daw - +          // Veure tots els avisos
§ 4

Comandes de gestió general

KEYS *                     // Llista totes les claus (evitar en producció)
SCAN 0                     // Alternativa segura a KEYS
TYPE mykey                 // Tipus de dada emmagatzemada

EXPIRE mykey 60            // La clau caduca en 60 segons
TTL mykey                  // Temps restant fins a caducar
PERSIST mykey              // Elimina la caducitat (fa la clau permanent)

DEL clau1 clau2 clau3      // Elimina múltiples claus
RENAME mykey nounom        // Reanomena la clau

// Exemple: sessió d'usuari amb caducitat de 30 minuts
SET sessio:user_42 "token_abc123"
EXPIRE sessio:user_42 1800  // 30 min = 1800 s
