Sistemes d'emmagatzematge de la informació

Criteri 1.1 — De fitxers a bases de dades

Abans de les bases de dades, la informació es guardava en fitxers. Cada aplicació tenia els seus propis fitxers, cosa que generava redundància, inconsistència i dificultat per compartir dades. Les bases de dades van sorgir per resoldre aquests problemes.

Tipus de fitxerAccésCaracterístiques
Seqüencial (pla)D'inici a fiSimple, però lent per a cerques. Cal llegir tot el fitxer per trobar un registre.
IndexatPer índex + seqüencialUn índex separat accelera les cerques. Exemple: ISAM (Indexed Sequential Access Method).
Accés directe (hash)Per adreça calculadaAccés molt ràpid per clau. Una funció hash transforma la clau en una adreça física.
Problemes dels sistemes de fitxers
Redundància: les mateixes dades es guarden en múltiples fitxers de departaments diferents.
Inconsistència: les còpies divergeixen (un departament actualitza el telèfon d'un client i l'altre no).
Dificultat de consulta: cal programar cada consulta específica, no hi ha un llenguatge estàndard.
Aïllament: les dades estan disperses en formats heterogenis, difícils de combinar.
Problemes de concurrència: dos usuaris modificant el mateix fitxer simultàniament poden corrompre les dades.

Tipus de bases de dades

Criteri 1.2 · 1.3 — Classificació per model i per ubicació

Les bases de dades es poden classificar des de dues perspectives: el model de dades que utilitzen (com s'organitza la informació internament) i la seva ubicació física (on resideix la informació).

Classificació per model de dades · Criteri 1.2

Relacional (SQL)

Organitza les dades en taules (relacions) amb files i columnes. Les taules s'interrelacionen mitjançant claus foranes. Exemple: Oracle, MySQL, PostgreSQL, SQL Server. El model predominant en sistemes transaccionals.

Orientada a objectes

Les dades s'emmagatzemen com a objectes, igual que en la programació orientada a objectes. Inclou herència, encapsulament i polimorfisme. Exemples: db4o, ObjectDB. Útil quan les dades tenen estructura complexa.

Jeràrquica

Les dades s'organitzen en una estructura d'arbre (pare-fill). Molt ràpida per a relacions 1:N predefinides, però rígida. Exemple: IMS d'IBM. Poc usada avui, però present en sistemes legacy.

En xarxa

Extensió del model jeràrquic: cada node pot tenir múltiples pares (grafs). Permet relacions M:N sense taules intermèdies. Exemple: CODASYL. Gairebé obsoleta avui.

NoSQL (no relacionals)

Família de models alternatius al relacional: clau-valor (Redis), documentals (MongoDB), columnars (Cassandra), grafs (Neo4j). Prioritzen escalabilitat i flexibilitat sobre la consistència estricta.

Relacional-objecte

Híbrid que afegeix al model relacional capacitats orientades a objectes: tipus definits per l'usuari, herència de taules, mètodes. Exemple: PostgreSQL en mode avançat, Oracle amb objectes.

Classificació per ubicació · Criteri 1.3

Centralitzada

Totes les dades resideixen en un únic servidor i s'hi accedeix de forma remota. Fàcil d'administrar i mantenir la consistència, però crea un únic punt de fallada i pot saturar-se amb molta càrrega.

Distribuïda

Les dades es reparteixen entre múltiples nodes interconnectats per xarxa. L'usuari la veu com una sola BD. Major disponibilitat i rendiment, però major complexitat de gestió i consistència.

En el núvol (Cloud)

La BD s'executa sobre infraestructura cloud (AWS RDS, Azure SQL, Google Cloud Spanner). Ofereix escalabilitat elàstica, alta disponibilitat i pagament per ús, sense gestionar el maquinari físic.

Paral·lela

Múltiples processadors treballen simultàniament sobre la mateixa BD en un sol sistema físic (shared-memory o shared-disk). Molt alt rendiment per a grans volums de dades. Exemple: Oracle RAC.

El Sistema Gestor de Bases de Dades

Criteri 1.4 · 1.5 · 1.6 — Funcions, components i classificació dels SGBD

Un SGBD (Sistema Gestor de Bases de Dades) és el programari que permet crear, mantenir i accedir a una base de dades de forma eficient, segura i multiusuari. Actua com a intermediari entre les aplicacions i les dades físiques.

Funcions principals d'un SGBD · Criteri 1.4
Definició: permet crear i modificar l'estructura (DDL). Manipulació: consultar, inserir, actualitzar i esborrar dades (DML). Control: gestió de permisos i privilegis (DCL). Integritat: imposa restriccions que garanteixen la validesa de les dades. Seguretat: controla qui pot fer què. Concurrència: permet l'accés simultani de múltiples usuaris sense conflictes. Recuperació: restaura la BD a un estat consistent després d'un error.

Arquitectura ANSI/SPARC — tres nivells · Criteri 1.5

Extern (vistes)Com cada usuari o aplicació veu les dades. Cada grup té la seva pròpia vista parcial de la BD.
ConceptualDescripció global de tota la BD: entitats, atributs, relacions, restriccions. Independitza les vistes dels detalls físics.
Intern (físic)Com s'emmagatzemen realment les dades al disc: estructures de fitxers, índexos, compressió. Transparent per a l'usuari.

Components d'un SGBD · Criteri 1.5

ComponentFunció
Motor de consultesAnalitza, optimitza i executa les sentències SQL.
Gestor de transaccionsGaranteix les propietats ACID (Atomicitat, Consistència, Aïllament, Durabilitat).
Gestor de bufferGestiona la memòria cau de pàgines de dades entre disc i memòria RAM.
Gestor de fitxersControla l'organització física de les dades al disc.
Gestor de seguretatAutenticació d'usuaris i control d'accés (privilegis, rols).
Catàleg / Diccionari de dadesMetadades: descripció de totes les taules, columnes, índexos, restriccions i usuaris.
Gestor de recuperacióDiari de transaccions (redo/undo log) per restaurar la BD després d'un error.

Bases de dades distribuïdes i fragmentació

Criteri 1.7 · 1.8 — Com es distribueixen les dades entre nodes

En una BD distribuïda, les dades es reparteixen físicament entre múltiples nodes, però el sistema ofereix una transparència de distribució: l'usuari interactua com si fos una única BD local.

Propietats desitjables · Criteri 1.7
Autonomia local: cada node pot operar de forma independent.
No dependència d'un node central: cap node és imprescindible per al funcionament global.
Operació contínua: el sistema segueix disponible si alguns nodes fallen.
Independència de localització: l'usuari no sap on resideix cada dada.
Independència de fragmentació: l'usuari no sap com s'han dividit les dades.

Tècniques de fragmentació · Criteri 1.8

Fragmentació horitzontal

Es divideix la taula per files (tuples). Cada fragment conté un subconjunt de les files que compleix una condició. Exemple: clients europeus al node de París, clients americans al de Nova York.

Fragmentació vertical

Es divideix la taula per columnes. Cada fragment conté totes les files però només alguns atributs (+ la clau primària per poder reconstruir). Útil per separar dades sensibles.

Fragmentació mixta

Combina les dues anteriors: primer es fragmenta verticalment i després cadascuna es fragmenta horitzontalment (o al revés). Permet una distribució molt flexible.

Replicació
A més de la fragmentació, es poden fer rèpliques: còpies d'un fragment en múltiples nodes. Augmenta la disponibilitat (si un node falla, un altre té la còpia) i el rendiment de lectura, però complica les actualitzacions (cal sincronitzar totes les còpies).

Legislació sobre protecció de dades

Criteri 1.9 — RGPD i LOPDGDD

Qualsevol sistema d'informació que emmagatzemi dades de persones físiques ha de complir la normativa vigent de protecció de dades. A Europa, el marc legal principal és el RGPD (Reglament General de Protecció de Dades).

NormaÀmbitPunts clau
RGPD
(UE 2016/679)		Tota la UE. Des de 2018.Principis: minimització, limitació de finalitat, exactitud, integritat. Drets: accés, rectificació, supressió (oblit), portabilitat, oposició. Multes fins a 20 M€ o el 4% de la facturació anual global.
LOPDGDD
(LO 3/2018)		Espanya. Adapta el RGPD.Concreta i amplia el RGPD per a l'ordenament jurídic espanyol. Regula la delegat de protecció de dades (DPD), el tractament de dades de menors (edat mínima 14 anys per consentir), drets digitals dels treballadors.
LSSI
(Llei 34/2002)		Serveis de la societat de la informació.Regula el comerç electrònic, les comunicacions comercials per email (spam), les cookies i les obligacions dels prestadors de serveis.
Implicacions pràctiques per a un dissenyador de BD
No emmagatzemar més dades personals de les estrictament necessàries (principi de minimització).
Definir períodes de retenció: quan s'han de suprimir les dades.
Implementar mesures tècniques: xifrat, control d'accés, pseudonimització.
Documentar el registre d'activitats de tractament.
Fer una avaluació d'impacte (EIPD) per a tractaments d'alt risc.

Big Data i intel·ligència de negocis

Criteri 1.10 — Quan les dades es fan massa grans per al model tradicional

El terme Big Data fa referència a conjunts de dades tan grans, variats o generats tan ràpidament que les eines tradicionals de bases de dades no poden gestionar-los eficientment. Es caracteritza per les 5 V:

Volum

Quantitat massiva de dades (terabytes, petabytes). Exemple: logs de servidor, dades de sensors IoT.

Velocitat

Generació i processament en temps real o gairebé real. Exemple: transaccions bancàries, streaming de sensors.

Varietat

Dades estructurades (taules), semiestructurades (JSON, XML) i no estructurades (imatges, vídeos, text lliure).

Veracitat

Qualitat i fiabilitat de les dades: les dades massives contenen soroll, errors i incertesa que cal gestionar.

Valor

L'objectiu final: extreure coneixement útil i presa de decisions a partir de totes les dades recollides.

Eines típiques

Hadoop (HDFS + MapReduce), Apache Spark, Hive, Kafka. Plataformes cloud: AWS EMR, Azure HDInsight, Google BigQuery.

Intel·ligència de negocis (BI)
La Business Intelligence agrupa tecnologies i processos per transformar dades en informació accionable. L'arquitectura típica és: Fonts de dades → ETL → Data Warehouse → OLAP → Informes i Dashboards.

ETL (Extract, Transform, Load): extreu dades de fonts heterogènies, les transforma i les carrega al DW.
Data Warehouse: base de dades optimitzada per a consultes analítiques (desnormalitzada, orientada a tema, variant en el temps).
OLAP (Online Analytical Processing): permet analitzar dades multidimensionalment (cubs de dades, drill-down, roll-up).
Data Lake: repositori de dades en brut en qualsevol format, que es processa quan es necessita (schema-on-read).
